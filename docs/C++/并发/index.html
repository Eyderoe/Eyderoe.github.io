并发

多线程和协程的一些东西

概念​

同步 / 异步​

并发​

逻辑上的同时发生。在宏观上是多个任务都在运行，微观实现上可能是单片机上的时间片轮转或者多核 CPU 的真同时运行

很显然的。线程满足该条件，多个线程同时运行，无论系统实际如何实现，宏观看就是多个程序在跑；协程也可以是，因为协程有能够暂停再恢复的特性

对比​

线程协程 (Coroutine)
操作系统内核（不可控）应用程序/运行时（可控）
抢占式（可能会在任何时刻被中断）协作式（必须手动 await 或 yield）
较大（通常 MB 级别栈空间）极小（通常 KB 级别）
复杂（需要锁 mutex、原子变量）简单（单线程下通常不需要锁）

线程​

非常重量级的东西，在 win 下的行为也和 unix 那桌不太一样

协程​

协程核心特性是能够满足暂停与恢复。不一定需要并发，不并发的一般是生成器，其他一般是长时间 IO 异步任务。这也导致了一个问题，异步任务整个的用户态调度需要自己写，不然只能等着库实现，比如 C++ 20 开始引入语法 23 才支持生成器 26 支持发送接收器

同步生成器​

生成器比较好理解，这是一段经典的生成斐波拉契数列的代码。不过不太像一般函数，返回值会被 generator 包裹，变成返回惰性求值的生成器，有点像 input_range，只能单向单次的遍历

std::generator<uint64_t> fibonacci () {
    uint64_t a = 0;
    uint64_t b = 1;
    while (true) {
        co_yield a;
        auto next = a + b;
        a = b;
        b = next;
    }
}

进入 while 后，每次遇到 co_yield 就会冻结状态并返回值，直到下一次的调用

异步任务​

代码总有些奇奇怪怪的 IO 任务，CPU 不参与但还是会等在这耗费大量时间。协程在这里就非常舒服，创建任务会把任务添加至事件循环，不会过多的占用 CPU 资源。比如 asio 的异步发送

asio::awaitable<void> send (const std::shared_ptr<std::string> data) {
    co_await infoSocket.async_send_to(asio::buffer(*data), infoEndpoint, asio::use_awaitable);
}

运行到 co_wait 就会冻结，返回，直到 async_send_to 完成后，才会继续执行。co_wait 在实现上为把当前代码包装为回调函数，注册给低层事件循环，内核完成任务后，再加入事件循环

对于 IO 任务来说，协程很好，可以避免等待，对于 CPU 任务来说，虽然协程本身作用不大，因为还是占用一样的时间。但在协程里面套一层多线程就可以很好的解决问题，既避免的线程的回调，又用上了协程的同步写法
